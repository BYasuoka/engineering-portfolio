Data Processing

LabVIEW to Stress vs. Strain Pipeline

This project automates a common lab bottleneck: taking raw LabVIEW measurement files and converting them into organized stress-strain outputs that are easier to analyze, compare, and share across multiple trials or specimen groups.

Inputs

Raw `.lvm` data plus specimen dimensions

The scripts walk through folders of LabVIEW Measurement files, pair them with either nominal or spreadsheet-based geometry values, and convert that raw data into usable engineering quantities.

Outputs

CSV cleanup, Excel workbooks, and averaged curves

The final output is not just one file. The pipeline creates intermediate CSVs, detailed Excel sheets for each trial, and cumulative summary workbooks that average rounded stress-strain data across repeated tests.

Processing Pipeline

How the scripts transform raw measurement files

1. Convert `.lvm` to CSV

The workflow opens each LabVIEW file, skips the header region, and writes the tab-delimited body into CSV format so it can be processed with pandas.

2. Remove nonessential rows

The scripts explicitly strip the first nine rows from the converted CSVs, creating “update” files that keep the later analysis focused on the useful signal.

3. Compute strain and stress

For every data row, the pipeline calculates global-displacement strain, extensometer strain, and engineering stress based on specimen length, area, and measured force.

4. Round and average trials

Strain values are rounded to the nearest 0.005 and grouped so multiple data points can be averaged into cleaner stress-strain curves.

Engineering Value

Turning repetitive lab cleanup into a repeatable workflow

This project is valuable because it reduces manual spreadsheet work and makes trial comparison more systematic. In lab settings, that kind of workflow automation can save a large amount of time and reduce mistakes when handling many files.

  • Supports multiple trials and naming conventions
  • Handles both nominal and spreadsheet-based specimen dimensions
  • Exports detailed and cumulative results for easier reporting

Opportunities To Improve

What I would refine in the next version

Path portability

The current scripts rely on hardcoded desktop paths, so moving to relative paths and a configuration file would make the pipeline easier to reuse on other machines.

Function boundaries

There is already good modular structure in places, but more separation between file I/O, calculations, and report generation would make testing easier.

Error handling

Additional checks for missing columns, inconsistent file naming, or empty trial sets would make the workflow more robust for classmates or teammates.

Automated plotting outputs

Saving plots directly to image files or report-ready figures would make the final step of sharing results even smoother.
